## فوری آغاز

1. GitHub پر اس ریپوزٹری کو **Fork** کریں۔
2. اپنے فورک کو مقامی طور پر **Clone** کریں:
   ```bash
   git clone https://github.com/AbdullahMalik17/malikclaw.git
   cd malikclaw
   ```
3. اپ اسٹریم ریموٹ ریپوزٹری شامل کریں:
   ```bash
   git remote add upstream https://github.com/AbdullahMalik17/malikclaw.git
   ```

## ڈویلپمنٹ ماحول کی ترتیب

### ضروریات

- Go 1.25 یا اس سے اعلی ورژن
- `make`

### بلڈ (Build)

```bash
make build       # بائنری فائل بنائیں (پہلے go generate چلے گا)
make generate    # صرف go generate چلائیں
make check       # مکمل پری سبمٹ چیک: deps + fmt + vet + test
```

### ٹیسٹ چلانا

```bash
make test                                    # تمام ٹیسٹ چلائیں
go test -run TestName -v ./pkg/session/      # ایک ٹیسٹ چلائیں
go test -bench=. -benchmem -run='^$' ./...  # بینچ مارک ٹیسٹ چلائیں
```

### کوڈ اسٹائل

```bash
make fmt   # کوڈ فارمیٹ کریں
make vet   # سٹیٹک تجزیہ
make lint  # مکمل لِنٹ چیک
```

تمام CI چیک پاس ہونے کے بعد ہی PR ضم (merge) کی جا سکتی ہے۔ کوڈ پش کرنے سے پہلے براہ کرم مقامی طور پر `make check` چلائیں تاکہ مسائل کا پہلے ہی پتہ چل سکے۔

## ترمیم جمع کرانا

### برانچ مینجمنٹ

ہمیشہ `main` برانچ سے نئی برانچ بنائیں، اور PR میں `main` کو ٹارگٹ برانچ کے طور پر رکھیں۔ براہ راست `main` یا کسی `release/*` برانچ میں کوڈ پش نہ کریں:

```bash
git checkout main
git pull upstream main
git checkout -b آپ_کی_برانچ_کا_نام
```

براہ کرم وضاحتی برانچ نام استعمال کریں، جیسے: `fix/telegram-timeout` ، `feat/ollama-provider` ، `docs/contributing-guide`۔

### Commit کے اصول

- واضح اور مختصر commit پیغامات انگریزی میں لکھیں۔
- حکمیہ جملے استعمال کریں: "Add retry logic" لکھیں، نہ کہ "Added retry logic"۔
- متعلقہ Issue کا حوالہ دیں: `Fix session leak (#123)`۔
- ہر commit صرف ایک کام کے لیے ہونا چاہیے۔
- چھوٹی صفائی یا ہجے کی اصلاح کے لیے، PR جمع کرانے سے پہلے انہیں ایک commit میں ضم کر دیں۔
- https://www.conventionalcommits.org/en/v1.0.0/ کے اصولوں پر عمل کریں۔

### اپ اسٹریم کے ساتھ ہم آہنگ رہنا

PR جمع کرانے سے پہلے، اپنی برانچ کو اپ اسٹریم `main` پر ری بیس (rebase) کریں:

```bash
git fetch upstream
git rebase upstream/main
```

## AI کی مدد سے تعاون

MalikClaw بڑی حد تک AI کی مدد سے تیار کیا گیا ہے، اور ہم اس طریقے کو مکمل طور پر اپناتے ہیں۔ تاہم، تعاون کرنے والوں کو AI ٹولز استعمال کرتے وقت اپنی ذمہ داریوں کا واضح ادراک ہونا چاہیے۔

### AI کے استعمال کا انکشاف ضروری ہے

ہر PR میں PR ٹیمپلیٹ کے **🤖 AI Code Generation** حصے کے ذریعے AI کی شرکت کا انکشاف کرنا ضروری ہے، جس کے تین درجات ہیں:

| درجہ | وضاحت |
|---|---|
| 🤖 مکمل طور پر AI کے ذریعے تیار کردہ | AI نے کوڈ لکھا، تعاون کرنے والے نے جائزہ لیا اور تصدیق کی |
| 🛠️ بنیادی طور پر AI کے ذریعے تیار کردہ | AI نے خاکہ بنایا، تعاون کرنے والے نے بڑی ترامیم کیں |
| 👨‍💻 بنیادی طور پر انسان کے ذریعے لکھا گیا | تعاون کرنے والے نے خود لکھا، AI نے صرف معمولی مدد کی یا استعمال نہیں ہوا |

ہم توقع کرتے ہیں کہ آپ اسے ایمانداری سے پُر کریں گے۔ تینوں درجات قابل قبول ہیں اور کوئی امتیازی سلوک نہیں کیا جائے گا—اہم بات تعاون کا معیار ہے۔

### جمع کرائے گئے کوڈ کی مکمل ذمہ داری آپ پر ہے

AI کے ذریعے کوڈ تیار کرنے کا مطلب یہ نہیں کہ آپ کی ذمہ داری ختم ہو گئی ہے۔ AI سے تیار کردہ کوڈ پر مشتمل PR جمع کرانے سے پہلے، آپ کو:

- تیار کردہ کوڈ کو **لائن بہ لائن پڑھنا اور سمجھنا** چاہیے۔
- **حقیقی ماحول میں ٹیسٹ** کرنا چاہیے (PR ٹیمپلیٹ میں ٹیسٹ ماحول کا حصہ دیکھیں)۔
- **سیکیورٹی کے مسائل چیک کریں** — AI ماڈلز ایسے کوڈ تیار کر سکتے ہیں جن میں سیکیورٹی کے خطرات ہوں (جیسے پاتھ ٹریورسل، انجکشن حملے، اسناد کا افشا وغیرہ)، براہ کرم احتیاط سے جائزہ لیں۔
- **درستگی کی تصدیق کریں** — AI کی منطق بظاہر درست لگ سکتی ہے لیکن حقیقت میں غلط ہو سکتی ہے، صرف گرامر نہیں بلکہ طرز عمل کی تصدیق کریں۔

اگر واضح ہو کہ تعاون کرنے والے نے AI سے تیار کردہ کوڈ کو نہیں پڑھا یا ٹیسٹ نہیں کیا، تو اس PR کو بغیر کسی جائزے کے بند کر دیا جائے گا۔

### AI سے تیار کردہ کوڈ کے معیار کے معیارات

AI سے تیار کردہ کوڈ اور انسان کے لکھے ہوئے کوڈ کے لیے **معیار کے مطالبات ایک جیسے ہیں**:

- تمام CI چیکس (`make check`) پاس ہونے چاہئیں۔
- Go کے معیاری طریقہ کار (idiomatic writing) کے مطابق ہونا چاہیے اور موجودہ کوڈ بیس کے اسٹائل سے ہم آہنگ ہونا چاہیے۔
- غیر ضروری پیچیدگی، ڈیڈ کوڈ یا ضرورت سے زیادہ ڈیزائن شامل نہ کریں۔
- جہاں ضروری ہو وہاں ٹیسٹ شامل یا اپ ڈیٹ کریں۔

### سیکیورٹی جائزہ

AI سے تیار کردہ کوڈ کو بہت احتیاط سے سیکیورٹی جائزے کی ضرورت ہوتی ہے۔ درج ذیل پہلوؤں پر خصوصی توجہ دیں:

- فائل پاتھ ہینڈلنگ اور سینڈ باکس سے فرار (پروجیکٹ کی تاریخ میں commit `244eb0b` ایک حقیقی مثال ہے)
- چینل ہینڈلرز اور ٹول امپلیمنٹیشن میں بیرونی ان پٹ کی تصدیق
- اسناد (credentials) یا کیز (keys) کا ہینڈلنگ
- کمانڈ ایگزیکیوشن (`exec.Command` وغیرہ)

اگر آپ کو یقین نہیں ہے کہ AI کا تیار کردہ کوڈ محفوظ ہے یا نہیں، تو PR میں اس کا ذکر کریں—جائزہ لینے والے فیصلہ کرنے میں مدد کریں گے۔

## Pull Request کا عمل

### PR جمع کرانے سے پہلے چیک کریں

- [ ] مقامی طور پر `make check` چلائیں اور تصدیق کریں کہ یہ پاس ہو گیا ہے۔
- [ ] PR ٹیمپلیٹ مکمل طور پر پُر کریں، بشمول AI انکشاف والا حصہ۔
- [ ] PR کی تفصیل میں متعلقہ Issue کا حوالہ دیں۔
- [ ] PR کو صرف ایک مقصد کے لیے رکھیں، غیر متعلقہ ترامیم کو مکس نہ کریں۔

### PR ٹیمپلیٹ کے حصوں کی وضاحت

PR ٹیمپلیٹ میں درج ذیل معلومات درکار ہیں:

- **تفصیل** — یہ تبدیلی کیا کرتی ہے اور کیوں کی گئی ہے؟
- **تبدیلی کی قسم** — بگ فکس، نیا فیچر، دستاویزات یا ری فیکٹرنگ۔
- **AI کوڈ جنریشن** — AI کی شرکت کا انکشاف (لازمی)۔
- **متعلقہ Issue** — اس PR سے حل ہونے والے Issue کا لنک۔
- **تکنیکی پس منظر** — حوالہ لنکس اور ڈیزائن کی وجوہات (صرف دستاویزات والی PR کے لیے چھوڑا جا سکتا ہے)۔
- **ٹیسٹ ماحول** — ٹیسٹنگ کے لیے استعمال ہونے والا ہارڈ ویئر، آپریٹنگ سسٹم، ماڈل/فراہم کنندہ اور چینل۔
- **تصدیقی ثبوت** — اختیاری لاگز یا اسکرین شاٹس جو ثابت کریں کہ تبدیلی موثر ہے۔
- **چیک لسٹ** — خود جائزے کی تصدیق۔

### PR کا سائز

براہ کرم چھوٹی اور آسانی سے جائزہ لینے کے قابل PR جمع کرانے کی کوشش کریں۔ 5 فائلوں میں 200 لائنوں کی تبدیلی والی PR، 30 فائلوں میں 2000 لائنوں کی تبدیلی والی PR سے کہیں زیادہ بہتر ہے۔ اگر آپ کا فیچر بڑا ہے، تو اسے چھوٹے منطقی حصوں میں تقسیم کرنے پر غور کریں۔

## برانچنگ حکمت عملی

### طویل مدتی برانچز

- **`main`** — فعال ڈویلپمنٹ برانچ۔ تمام فیچر PRs کا ٹارگٹ `main` ہوتا ہے۔ یہ برانچ محفوظ ہے: براہ راست پش ممنوع ہے، ضم کرنے سے پہلے کم از کم ایک مینٹینر کی منظوری ضروری ہے۔
- **`release/x.y`** — مستحکم ریلیز برانچ، جب کوئی ورژن ریلیز کے لیے تیار ہو تو اسے `main` سے الگ کیا جاتا ہے۔ ان برانچوں کی حفاظت کا درجہ `main` سے زیادہ ہے۔

### `main` میں ضم ہونے کی شرائط

PR ضم ہونے کے لیے درج ذیل تمام شرائط کا پورا ہونا ضروری ہے:

1. **تمام CI پاس ہوں** — تمام GitHub Actions ورک فلوز سبز ہونے چاہئیں۔
2. **جائزہ لینے والے کی منظوری** — کم از کم ایک مینٹینر نے PR منظور کی ہو۔
3. **کوئی حل طلب کمنٹ نہ ہو** — جائزے کے دوران اٹھائے گئے تمام نکات حل ہو چکے ہوں۔
4. **PR ٹیمپلیٹ مکمل ہو** — بشمول AI انکشاف اور ٹیسٹ ماحول کی معلومات۔

### کون ضم کر سکتا ہے؟

صرف مینٹینرز ہی PR ضم کر سکتے ہیں۔ تعاون کرنے والے اپنی PR خود ضم نہیں کر سکتے، چاہے ان کے پاس رائٹ پرمیشن کیوں نہ ہو۔

### ضم کرنے کی حکمت عملی

`main` کی تاریخ کو واضح اور پڑھنے کے قابل رکھنے کے لیے، ہم زیادہ تر PRs کے لیے **Squash Merge** استعمال کرتے ہیں۔ ہر ضم شدہ PR ایک علیحدہ commit بن جاتی ہے جس میں PR نمبر شامل ہوتا ہے، مثال کے طور پر:

```
feat: Add Ollama provider support (#491)
```

اگر کسی PR میں متعدد آزاد اور واضح commit ہوں جو ایک مکمل کہانی بیان کریں، تو مینٹینر عام merge بھی استعمال کر سکتا ہے۔

### Release برانچ

جب کوئی ورژن تیار ہو جاتا ہے، تو مینٹینر `main` سے `release/x.y` برانچ بناتا ہے۔ اس کے بعد:

- **نئے فیچرز شامل نہیں کیے جاتے۔** ریلیز برانچ بننے کے بعد، اس میں کوئی نیا فیچر شامل نہیں کیا جاتا۔
- **سیکیورٹی فکسز اور اہم بگ فکسز شامل کیے جاتے ہیں۔** اگر `main` پر کوئی فکس سیکیورٹی، ڈیٹا ضائع ہونے یا کریش سے متعلق ہو، تو مینٹینر اسے متعلقہ `release/x.y` برانچ میں شامل (cherry-pick) کرے گا اور پیچ ورژن جاری کرے گا۔

اگر آپ کو لگتا ہے کہ `main` پر کوئی فکس کسی ریلیز برانچ میں جانا چاہیے، تو براہ کرم PR کی تفصیل میں اس کا ذکر کریں یا الگ سے Issue بنائیں۔ حتمی فیصلہ مینٹینرز کریں گے۔

ریلیز برانچ کا تحفظ `main` سے زیادہ ہے، کسی بھی صورت میں براہ راست پش کی اجازت نہیں ہے۔

## کوڈ کا جائزہ

### تعاون کرنے والوں کے لیے تجاویز

- مناسب وقت میں ریویو کمنٹس کا جواب دیں۔ اگر مزید وقت چاہیے تو مطلع کریں۔
- فیڈ بیک کے جواب میں PR اپ ڈیٹ کرتے وقت مختصر وضاحت دیں (مثلاً: "تجویز کے مطابق `sync.RWMutex` استعمال کیا گیا ہے")۔
- اگر آپ کسی فیڈ بیک سے متفق نہیں ہیں، تو شائستگی سے اپنی وجہ بیان کریں—جائزہ لینے والے سے بھی غلطی ہو سکتی ہے۔
- جائزہ شروع ہونے کے بعد force push نہ کریں—اس سے تبدیلیوں کو ٹریک کرنا مشکل ہو جاتا ہے۔ اضافی commit استعمال کریں، مینٹینر ضم کرتے وقت squash کر دے گا۔

### جائزہ لینے والوں کے لیے تجاویز

جائزے کی ترجیحات:

1. **درستگی** — کیا کوڈ وہی کرتا ہے جس کا دعویٰ کیا گیا ہے؟ کیا کوئی خاص صورت (edge case) رہ گئی ہے؟
2. **سیکیورٹی** — AI سے تیار کردہ کوڈ، ٹول امپلیمنٹیشن اور چینل ہینڈلرز پر خاص توجہ دیں۔
3. **آرکیٹیکچر** — کیا امپلیمنٹیشن موجودہ ڈیزائن کے مطابق ہے؟
4. **سادگی** — کیا کوئی آسان حل موجود ہے؟ کیا غیر ضروری پیچیدگی شامل کی گئی ہے؟
5. **ٹیسٹنگ** — کیا تبدیلی کے لیے ٹیسٹ موجود ہیں؟ کیا پرانے ٹیسٹ اب بھی کارآمد ہیں؟

براہ کرم تعمیری اور مخصوص فیڈ بیک دیں۔ "اگر دو گوروٹین ایک ساتھ اس فنکشن کو کال کریں تو ریس کنڈیشن ہو سکتی ہے، یہاں ایک میوٹیکس شامل کرنے کی ضرورت ہے" کہیں زیادہ بہتر ہے بجائے اس کے کہ "یہاں کوئی مسئلہ لگتا ہے"۔

## مواصلاتی ذرائع

- **GitHub Issues** — بگ رپورٹس، فیچر کی تجاویز، ڈیزائن پر بحث۔
- **GitHub Discussions** — عام سوالات، آئیڈیاز کا تبادلہ، کمیونٹی ڈسکشن۔
- **Pull Request کمنٹس** — مخصوص کوڈ سے متعلق فیڈ بیک۔
- **Wechat اور Discord** — جب آپ کی کم از کم ایک PR ضم ہو جائے گی، ہم آپ کو ڈویلپر گروپ میں شامل ہونے کی دعوت دیں گے۔

کسی بھی شک کی صورت میں، پہلے Issue بنا کر بحث کریں، پھر کوڈ لکھنا شروع کریں۔ اس سے فضول محنت سے بچا جا سکتا ہے۔
